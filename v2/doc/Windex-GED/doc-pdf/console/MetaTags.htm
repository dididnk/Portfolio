Indexation des champs

Liste des champs ?

Dans les documents indexés, il existe des informations internes et non visibles par l'utilisateur. Suivant le type de document, elles sont stockées sous différentes formes. Par exemple, dans les documents de type HTML, elles sont enregistrées via la balise HTML META dont la syntaxe simplifiée est :

<META name="nom_de_la_META" content="valeur">
On trouve cette balise généralement dans l'entête du document HTML, au niveau de la balise HEAD. Les balises META contiennent des méta-informations, c'est-à-dire des informations de service, concernant le contenu de la page et non sa présentation.
Il n'y a pas de règles à proprement parlé sur l'usage des META. Ce sont des zones d'information libres. Certaines sont codifiées mais vous pouvez créer les vôtres.

Exemples :

<META name="Author" content="Association des Amis de la Poésie">
<META name="Classement" content="texte">
<META name="keywords" content="écrivain;français;poésie;texte;recueil">
<META name="description" content="Recueil de textes divers d'Alphonse Daudet"> 

 En règle générale, un champ peut contenir plusieurs valeurs : il suffit de les séparer par des point-virgules. Les exceptions sont précisées ci-dessous.

Les fichiers PDF possèdent un titre et un résumé. Ces informations sont automatiquement indexées par Windex Server.
Le titre est référencé dans le champ title. Les autres informations du document PDF sont enregistrées dans des champs du même nom.
Exemple :
<META name="CreationDate" content="D:19970822162327">
<META name="Creator" content="lp3820 v2.5">
<META name="ModDate" content="D:19970826122514">
<META name="Producer" content="Acrobat Distiller 3.0 for Windows">
<META name="Subject" content="Documentation">

Nom du champ

Dans le cas d'un document HTML, il s'agit du nom de la balise META.
Par convention :

  • utilisez uniquement des lettres et des chiffres; Windex Server ne fait pas de différence entre les majuscules et les minuscules
  • n'utilisez pas de caractères accentués, d'espaces ou tout autre caractère. Cependant, vous pouvez utiliser le caractère souligné "_".
Sauf exception, si un même champ apparaît plusieurs fois dans un document, c'est le contenu de sa dernière occurrence qui prévaut.

Si le nom du champ commence et finit par le caractère '$', alors sa valeur est calculée par le macro langage de Windex Server.
Exemple "$@DocRootname$" retourne le nom du document (sans son chemin)
Windex Server considère ces champs comme des champs virtuels.

Prenons comme exemple l'URL d'un document indexé :

http://mon_site:8080/documents/personnel/exemples/fichier.htm

Ces noms sont très pratiques car ils permettent, par exemple, de faire une recherche sur des documents :

  • situés dans un dossier particulier,
  • sous un certain format : PDF, texte, ...
  • sur un site en particulier
Notez bien que les noms de ces champs doivent être encadrés par un $ lors de leur utilisation.

L'utilisation des champs virtuels apportent de la puissance au moteur de recherche (utilisation de catégories virtuelles).

Il est possible d'utiliser le macro langage pour écrire des formules complexes.

Exemple :

    L'adresse de départ d'indexation est :

      http://www.mon_site.fr/documents

    Dans le dossier documents, on retrouve deux sous-dossiers : prive et public

      http://www.mon_site.fr/documents/public
      http://www.mon_site.fr/documents/prive

    On crée un champ qui indique si un document indexé appartient à l'arborescence privée ou publique.
    Ce champ sera de type catégorie et aura pour nom de champ la formule suivante :

      $@Mid( @DocPath('2'),@Length(@DocPath('1')))$

    Pour l'URL d'un document situé dans public ou prive :

      @DocPath('2') retourne soit /documents/public/ soit /documents/prive/

      @DocPath('1') retourne /documents/

      La fonction @Mid retourne alors, soit public/ soit prive/.

Nom associé

Le nom associé correspond au nom du champ stocké dans la base de données de Windex Server. Ce nom doit contenir uniquement des caractères alphanumériques et les caractères "_" et ".", sauf exception :

  • pour les champs de type titre : le nom associé est obligatoirement @Title.

Ce nom @Title est utilisé également dans le langage de recherche pour effectuer des recherche sur le titre ainsi défini.

Type

Windex Server associe un type à chaque champ.

Aucun contrôle n'est effectué entre le type d'un champ et le contenu trouvé dans un document.

Il existe 5 types :

TypeDescription
Texte

Son contenu est indexé en plein texte.

En général son contenu n'est pas visible par l'utilisateur.

Numérique

Les valeurs de type numérique sont considérées comme des nombres entiers compris entrte -2147483648 et +2147483647.
La recherche sur les nombres pourra se faire avec des opérateurs de comparaison.

Un champ de type numérique doit contenir une seule valeur.
Date

Le champ peut contenir des dates au format jj/mm/aaaa ou aaaa/mm/jj.
Par exemple, la date du 8 juillet 1978 peut être stockée sous le format 08/07/1978 ou 1978/07/08.

Un champ de type date doit contenir une seule valeur.
Titre

Le titre des documents HTML est par défaut indiqué par la balise <TITLE>. On peut utiliser également un champ dont la valeur représentera le titre.

La balise <TITLE> est utilisée par Windex Server pour générer le titre du document présenté dans la page des résultats.

Le nom associé au champ de type titre est imposé : @Title. Si vous tentez de le forcer à une autre valeur, celle-ci sera automatiquement remplacée.

Tous les champs de type titre sont concaténées pour constituer le titre final du document. Les différentes valeurs sont séparées par le caractère de saut à la ligne (\n).
Pour afficher dans la page des résultats chaque élément du titre sur une ligne distincte, utilisez la fonction @HtmlEncode().
Catégorie

Vous pouvez regrouper des documents au sein de catégories. Ce regroupement permet d'affiner les résultats en recherchant les documents liés à un thème donné.

Exemple :

    Vous disposez de documents résumant des grands classiques du cinéma. Vous souhaitez les regrouper en fonction du genre : horreur, humour, drame, amour, historique, classique.

    Vous décidez de créer la catégorie "genre". Afin de faciliter la saisie et de diminuer les risques de fautes de frappes, vous établissez la correspondance suivante :

Genre Codification
horreur ho
humour hu
drame dm
amour am
historique hi
classique cls

      Ainsi, pour un document sur un film historique, vous pourrez ajouter dans le document HTML, la ligne suivante :

<META name="genre" content="hi">

      Pour un document sur un film d'horreur parodique, on pourra considérer que le document appartient au genre humour et horreur :

<META name="genre" content="ho;hu">

Un champ de type catégorie peut contenir plusieurs valeurs : il suffit de les séparer par des points virgules.

    En utilisant des champs virtuels, il est possible de créer des catégories également virtuelles.

    Exemples :

      1/ On dispose d'un site intranet, comportant deux sous-dossiers :

      • private : contenant des documents privés (http://www.site.fr/private)
      • public : contenant des documents public (http://www.site.fr/public)

      Chacun de ces dossiers contient à son tour des sous-dossiers et des documents.

      Concrètement, on pourra créer le champ suivant :

      • Nom du champ : $@DocPath('1')$
      • Nom associé : Classement
      • Type : Catégorie

      Pour chaque document situé dans le dossier private ou un sous-dossier de celui-ci, cette catégorie aura pour valeur "private". En effet, le champ virtuel @DocPath('1') pour l'URL de ces documents retournera /private/.

      Pour chaque document situé dans le dossier public ou un sous-dossier de celui-ci, cette catégorie aura pour valeur "public". En effet, le champ virtuel @DocPath('1') pour l'URL de ces documents retournera /public/.

      On obtiendra ainsi une catégorie permettant à l'utilisateur de rechercher des documents soit privés soit publics.

      2/ Une société conserve en intranet les devis de ses clients et de ses fournisseurs. Les devis acceptés sont classés séparément des devis refusés pour chacun des cas :

      On obtient ainsi les URLs suivants :

      http://www.lasociete.fr/devis_clients/refuse/
      http://www.lasociete.fr/devis_clients/accepte/
      http://www.lasociete.fr/devis_fournisseurs/refuse/
      http://www.lasociete.fr/devis_fournisseurs/accepte/

      On pourra créer le champ suivant :

      • Nom du champ : $@DocPath('2')$
      • Nom associé : Type_Devis
      • Type : Catégorie
Dossier Devis destiné au Etat du devis Valeur de @DocPath('2')
http://www.lasociete.fr/devis_clients/refuse/ Client Refusé /devis_clients/refuse/
http://www.lasociete.fr/devis_clients/accepte/ Client Accepté /devis_clients/accepte/
http://www.lasociete.fr/devis_fournisseurs/refuse/ Fournisseur Refusé /devis_fournisseurs/refuse/
http://www.lasociete.fr/devis_fournisseurs/accepte/ Fournisseur Accepté /devis_fournisseurs/accepte/

      Cette catégorie, à l'instar de l'exemple précédent, permettra à l'utilisateur de rechercher directement un devis parmi ceux destinés aux fournisseurs et qui ont été refusés.

Priorité

La priorité permet d'influencer le calcul de la pertinence d'une réponse. Windex Server évalue la pertinence d'un document en fonction de la localisation des mots trouvés dans le document.

La priorité permet de modifier l'ordre de prise en compte (sauf pour le corps du document).

La priorité est comprise entre 1 (valeur par défaut et priorité la plus faible) et 127 (priorité la plus forte).

Les scripts

Les zones "Script à exécuter avant d'indexer le document" et "Script à exécuter après l'indexation du document" exécute un script de macro langage respectivement avant et après l'indexation d'un document

Avant l'indexation, le macro langage ne peut pas accéder aux informations internes du document (ex: résumé, titre, propriétés ...).
Dans la zone "Script à exécuter avant d'indexer le document" vous pouvez vous connecter à une base de donnée SQL, y extraire une "note" la stocker dans une variable (ex par : @Set('note', maNoteSQL) ) et l'utiliser dans un champ pour l'indexer.
Vous pouvez aussi utiliser un script pour afficher dans la console (ou le log applicatif) une trace sur le document indexé :

@println("Url indexé : " , @docurl)
